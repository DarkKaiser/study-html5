W3C - "The WebSocket API" 번역

본 문서에는 직역이 아닌 의역 부분이 많이 있습니다. 따라서 원문과 표현이 다를 수 있으므로 이점에 대해서는 양해 부탁드립니다. 또한 이 한국어 번역은 여러가지 번역상의 오류가있을 수 있습니다. 따라서, 내용에 대해 어떠한 보증도 하지 않으므로, 정확성을 요구하는 경우에는 반드시 원문을 참조하시길 권해드립니다. 본 문서의 사용으로 인한 이용자가 입는 손해에 대해서는 책임을 지지 않습니다. 만약 오류 등을 발견하시면 댓글로 알려주시면 감사하겠습니다.

개요

Web Sockets은 JavaScript에서 서버와 양방향 통신 메시지 교환을 제공하는 API입니다. XHR에서는 실현될 수 없었던 같은 실시간 상호 작용을 실현할 수 있기 때문에 지금까지에는 없었던 새로운 웹 응용 프로그램을 예감 수 있습니다.

현재 W3C의 HTML5 스펙에서 분리되어 버렸습 니다만, 차세대 웹 애플 리케이션의 핵심을 담당하는 기술로서 기대되고 있습니다.

1 소개

이 섹션은 비 규정입니다.

웹 애플 리케이션에서 서버측 프로세스와 양방향 통신을 할 수 있도록 본 사양은  WebSocket  인터페이스를 도입합니다.

이 인터페이스에서 가상 네트워크에 직접 액세스할 수 없습니다. 예를 들어,이 인터페이스를 사용하여 IRC 클라이언트를 구현할 수 없을 것입니다. 이렇게하기 위하여는 메시지를 독자 서버에 통해 프록시해야 안됩니다.

2 준수 요구 사항

이 사양은 각 절 아닌 규정이라고 기재되어 있으면, 그 부분에있는 도표와 예제 및주의 사항은 모두 숨기기 규정입니다. 다른 모든 규정입니다.

이 사양 규정 부분에있는 "해야한다 (MUST)", "하지 말라 (MUST NOT)", "해야 할 필수적인 (REQUIRED)", "해야한다 (SHOULD) ""해야하지 (SHOULD NOT) ","권장 (RECOMMENDED) ','해도 상관 없다 할 수있다 (MAY) ","옵션 (OPTIONAL) "라는 키워드는 RFC2119 규정대로 해석합니다. 이 사양은 읽기 쉽게하기 위해이 기간 모두 대문자로 표현하고 있습니다. [RFC2119]

알고리즘의 일부로 규정에서 언급된 요구 사항 ( "strip any leading space characters"와 "return false and abort these steps")는 그 알고리즘의 소개로 사용되는 키워드 ( "must", "should" "may"등)의 의미로 해석해야합니다.

몇 가지 준수 요구 사항은 속성, 메서드, 개체의 요구 사항으로 언급되고 있습니다. 이러한 요구 사항은 사용자 에이전트의 요구 사항으로 해석해야합니다.

알고리즘이나 특정 단계로 언급된 준수 요구 사항은 최종적인 결과가 같은 한 어떤 방식으로 구현해도 상관 없습니다. (특히,이 스펙에 정의된 알고리즘은 가독성을 고려한 것으로, 성능을 고려하고있는 것은 없습니다.)

이 사양에 정의된 규격 클래스는 사용자 에이전트뿐입니다.

사용자 에이전트는 다른 입력에 대한 구현상의 제한을 자발적으로 부과해도 상관 없습니다. 예를 들어, DoS 공격을 방지하기 위해 또는 메모리 부족을 방지하기와 플랫폼의 제한에 대한 해결 방법 등입니다.

기능 지원을 무효로 할 경우 (보안 문제를 줄이기 위해 비상 수단으로 또는 개발을 보조하기 위해 또는 성능 이유)는 사용자 에이전트는 그 기능을 전혀 지원 없는 것처럼, 그리고 그 기능이 사양에는 없었던 것처럼 대접해야 안됩니다. 예를 들어, 특정 기능을 Web IDL 인터페이스의 속성을 통해 접근할 수있다면 그 속성 자체는 그 인터페이스를 가진 개체에서 제외됩니다. 그 특성을 개체에 보관하면서 null을 반환하거나 예외를 던지도록했다만으로는 충분하지 않습니다.

3 용어의 정의

" Foo 오브젝트 "라고하면, Foo 는 실질적으로 인터페이스를 가리키는 것입니 다만, " Foo 인터페이스를 구현하는 개체 "보다는 정확한 표현 대신 사용될 수 있습니다.

DOM이라는 용어는 웹 애플 리케이션 스크립트에서 사용할 수 있도록하는 API 집합을 가리키는 데 사용되지만 반드시 DOM Core 사양에 정의된 것과 같은 Document 객체와 다른 Node 객체가 실제로 존재 하는 것을 의미하는 것은 아닙니다. [DOMCORE]

IDL 속성은 그 값 (예를 들어, 웹 제작자 측의 스크립트) 배출된 때 "포획된다"고 말하고, 그것에 새로운 값이 할당되는 경우에는 "세트된다"고 말합니다.

4 WebSocket 인터페이스

 WebSocket Implements EventTarget ;

WebSocket ( url , protocols ) 생성자는 1 개 2 개의 인수를 취합니다. 제 1 인수 url 에는 어디에 연결할지 여부를 나타내는 URL을 지정합니다. 제 2 인수 protocols , 만약 지정한다면, 문자열, 또는 문자열의 배열 중 하나를 지정합니다. 만약 문자열이 지정되면, 그 문자 열만 포함하는 배열과 동일합니다. 생략되면 빈 배열과 동일합니다. 배열의 각 문자열은 하위 프로토콜 이름입니다. 서버가 이러한 하위 프로토콜 중 하나를 선택했다고보고한다면 그 연결만 설정됩니다. 서브 프로토콜 이름은 모두 비어 있지 않은 ASCII 캐릭터 라인이 아니면 안됩니다. 그러나 제어 문자와 공백 넣을 수 없습니다 (즉, U +0021에서 U +007 E까지의 문자만입니다.)

WebSocket () 생성자가 호출되면 사용자 에이전트는 다음 단계를 실행해야 안됩니다 :

  1. url 인수에서 WebSocket의 URL 구성 요소를 파싱하여 host , port , resource name , secure을 가져옵니다. 이것이 실패하면 SYNTAX_ERR 예외를 던지고 이러한 절차를 중단합니다. [WSP]

  2. port 가 사용자 에이전트가 액세스를 차단하도록 설정되어있는 포트면 SECURITY_ERR 예외를 던집니다. (일반적으로 사용자 에이전트는 SMTP와 같은 well - known 포트에 대한 액세스를 차단합니다)

    포트 80 및 443에 대한 액세스는 secure 가 false하지만 port 가 443 또는 secure 가 true하지만port 80 같은있을 수없는 경우도 포함하여 차단해야하지 않습니다.

  3. protocols 이 존재하지 않는다면, protocols 빈 배열합니다.

    그렇지 않으면, protocols 이 존재하고 그것이 문자열이라면 protocols 를 문자열만으로 구성되는 배열로 바꿉니다.

  4. protocols 같은 값이 포함되어 있고, U +0021 미만 또는 U +007 E보다 큰 Unicode 코드 포인트를 수반하는 문자를 포함한 값이 하나라도 있으면 (즉, 공백 문자와 인쇄 가능한 ASCII 문자가 아닌 문자) SYNTAX_ERR 예외를 던지고 이러한 절차를 중단합니다.

  5. origin 을 WebSocket () 생성자를 호출한 스크립트 origin의 ASCII 직렬화합니다. 그러나 소문자로 변환된 것입니다.

  6. 새로운 WebSocket 개체를 반환 백그라운드에서 이러한 단계를 계속합니다 (스크립트를 차단하지 않습니다.)

  7. 호스트 host 에 포트 port 에서 (지정되면), origin 에서 플래그 secure 을 지정하고 자원 이름resource name 을 지정 프로토콜 목록 (비어있을 수도 있음) protocols 을 지정하고 defer cookies 세트를 지정하고 WebSocket 연결을 설정합니다. [WSP]

    만약 "WebSocket 연결 설정"알고리즘이 실패하면 "WebSocket 연결을 포기"알고리즘이 호출됩니다. 이것은 아래과 같이 "WebSocket 연결 끄기"알고리즘을 호출한 다음 "WebSocket 연결이 끊긴"을 설정하고 close 이벤트를 발표합니다.

스크립트 전역 개체가 Window 개체와 WorkerUtils 인터페이스를 구현하는 개체 중 하나가 될 때이 생성자를 볼 수 있도록해야 안됩니다.

url 속성은 생성자에 인수로 전달된 URL의 해결 결과를 돌려주지 않으면 안됩니다. (그것이 어디에서 상대 기준으로 해결된 지에 대해서는 문제가 없습니다. 왜냐하면 그것은 이미 절대 URL이라고 알고 있기 때문입니다.)

readyState 속성은 연결 상태를 나타냅니다. 이것은 다음의 값을 가지고 있습니다 :

CONNECTING (숫자 0)
연결이 아직 확립되어 있지 않다.
OPEN (숫자 1)
WebSocket 연결이 설정되고 통신이 가능.
CLOSING (숫자 2)
연결이 끊어 핸드 셰이크 중.
CLOSED (숫자 3)
연결이 끊어진 또는 열 수 없었다.

객체 생성시 그 readyState 는 CONNECTING (0)이 설정되어야 안됩니다.

the protocol 속성은 처음에는 빈 문자열을 반환해야 안됩니다. WebSocket 연결이 설정되면, 그 값은 아래와 같이 바뀝니다.

protocol 속성은 서버에서 선택된 하위 프로토콜이 있으면, 그것을 돌려줍니다.그것은 하위 프로토콜 협상을 수행하기 위해 생성자의 두번째 파라미터의 배열 형식과 함께 사용될 수 있습니다.

send ( data ) 메서드는 연결을 사용하여 데이터를 전송합니다. readyState 속성이CONNECTING 면 INVALID_STATE_ERR 예외를 발표해야 안됩니다. 그렇지 않으면, 만약 인수 data 비 서로 게이트 쌍이 있으면 SYNTAX_ERR 을 발표해야 안됩니다. 만약 연결이 설정되고 그 문자열이 아닌 서로 게이트 쌍이 아니라 WebSocket 절단 핸드 셰이크가 아직 시작되지 않은 경우 사용자 에이전트는 WebSocket을 사용하여 data 를 전송해야 깨지는 않습니다. 만약 버퍼 필요가 있었더라도 버퍼가 가득했다 등의 이유로 데이터가 전송되지 못한 경우 사용자 에이전트는 WebSocket 연결을 절단해야 안됩니다.이 메서드가 호출되고 예외가 발생하지 않으면 인수를 UTF - 8로 표현하는데 필요한 바이트 수만큼bufferedAmount 속성을 늘려야 안됩니다. [WSP]

close () 메서드는 다음 목록에서 처음 일치하는 단계를 실행해야 안됩니다 :

readyState 속성이 CLOSING (2) 또는 CLOSED (3) 상태에있는 경우

아무것도하지 않습니다.

연결이 이미 꺼져하려고하는 동안 또는 이미 꺼져 있습니다. 그래도 한 경우 아래와 같이 결국에는 close 이벤트이 발표됩니다.

WebSocket 연결이 아직 확립되어 있지 않은 경우 [WSP]

WebSocket 연결을 포기하고 readyState 속성 값을 CLOSING (2)로 설정합니다. [WSP]

"WebSocket 연결을 포기"알고리즘 "WebSocket 연결 끄기"알고리즘을 호출합니다. 이것은 다음 "WebSocket 연결이 끊긴"을 설립하는 것으로, 아래와 같이 , close 이벤트를 발표합니다.

WebSocket 절단 핸드 셰이크가 아직 시작되지 않은 경우 [WSP]

WebSocket 절단 핸드 셰이크를 시작하여 readyState 속성 값을 CLOSING (2)로 설정합니다. [WSP]

"WebSocket 절단 핸드 셰이크를 시작하는"알고리즘은 마지막으로 "WebSocket 연결 끄기"알고리즘을 호출합니다. 이것은 다음 "WebSocket 연결이 끊긴"을 설치하고 아래와 같이 , close 이벤트를 발표합니다.

그렇지 않으면

readyState 속성 값을 CLOSING (2)로 설정합니다.

WebSocket 절단 핸드 셰이크가 시작되었습니다. 그리고 마지막으로 "WebSocket 연결 끄기"알고리즘이 호출됩니다. 이것은 아래과 같이 "WebSocket 연결이 끊긴"을 설치하기 때문에, close 이벤트가 발됩니다.

bufferedAmount 속성은 send () 를 사용하여 대기열되었지만 이벤트 루프 작업의 실행을 시작하는 마지막 시점에서 아직 네트워크에 전송되지 않은 UTF - 8 텍스트의 바이트 수를 반환해야한다 않습니다.(따라서, 이것은 사용자 에이전트가 스크립트 실행과 비동기적으로 텍스트를 전송할 수 있는지 여부에 관계없이 현재 작업 중에 전송된 텍스트를 포함합니다.) 이것은 프로토콜에 의해 발생하는 프레임의 오버헤드 및 운영 시스템 및 네트워크 하드웨어에 의해 수행되었다 버퍼링을 포함하지 않습니다. 연결이 끊어진 경우이 속성 값은 send () 메소드가 호출될 때마다 증가됩니다 (이 수치는 연결이 끊어도 제로로 리셋되지 않습니다).

이 간단한 예제에서는 네트워크가 견딜 수있는 경우 50ms마다 자주 1 회 업데이 트를 보낼 수 있는지, 또는, 만약 너무 빨리면 어떤 빈도면 네트워크 견딜 수있는 지 여부를 확인하기 위해bufferedAmount 속성 가 사용되고 있습니다.

var socket = new WebSocket ( 'ws : / / game.example.com : 12010/updates'); socket.onopen = function () {setInterval (function () {if (socket.bufferedAmount == 0) socket.send ( getUpdateData ());}, 50);};

네트워크 품는 이상의 빈도로 데이터를 보내지 않고도 bufferedAmount 속성을 사용하면 네트워크를 포화 수 있습니다. 그러나 그 속성의 값을 항상 주의깊게 모니터링해야 안됩니다.

WebSocket 인터페이스를 가진 모든 개체 IDL 속성으로 지원해야 말라 이벤트 핸들러는 다음과 같습니다 :

이벤트 핸들러이벤트 핸들러의 이벤트 유형
onopen open
onmessage message
onerror error
onclose close

5 프로토콜의 의견

WebSocket 연결이 설정 되면 사용자 에이전트는 먼저 readyState 속성 값을 OPEN (1)로 변경하는 작업을 대기해야 안됩니다. 연결이 설정했을 때, list of cookies 모인 cookie을 적용합니다. 선택된 WebSocket 하위 프로토콜이 있으면 protocol 속성의 값을 변경합니다. 그리고 WebSocket 개체open 이라는 간단한 이벤트를 발표합니다. [WSP]

WebSocket 메시지를 텍스트 데이터로 수신 일단 사용자 에이전트는 MessageEvent 인터페이스를 사용하는 이벤트를 생성해야 안됩니다. 이것은 이벤트 이름을 message 에 버블 링하지 않고 취소하지 못하고, 디폴트 액션을 가지지 않고, 그 data 속성은 data 에 설정되고 readyState 속성 값이 OPEN (1) 또는 CLOSING (2) 여부를 확인 작업을 큐잉하고 그렇다면, WebSocket 개체에서 이벤트를 발표합니다.[WSP]

WebSocket 오류를 감지 일단 사용자 에이전트는 readyState 속성 값이 OPEN (1) 또는 CLOSING(2)인지를 확인하는 작업을 대기해야 안됩니다. 그렇다면, WebSocket 개체 error 라는 간단한 이벤트를 발표합니다. [WSP]

WebSocket 절단 핸드 셰이크가 시작 되면 사용자 에이전트는 readyState 속성 값을 CLOSING (2) 변경 작업을 대기해야 안됩니다. ( close () 메소드가 호출되면이 작업을 수행할 때 이미 readyState속성 값은 CLOSING (2)로 설정되어 있습니다. ) WSP]

WebSocket 연결이 끊어 되면 가능한 한 완전하게, 사용자 에이전트는 CloseEvent 인터페이스를 사용하는 이벤트를 생성해야 안됩니다. 이것은 이벤트 이름 close 으로 버블링하지 않고 취소하지 못하고, 디폴트 액션을 가지지 않고, 연결이 완전히 차단되어있다면, 그 wasClean 속성은 true로 설정되고 그렇지 않으면 false로 설정 되어 그 code 속성은 WebSocket 연결 끊기 이유로 세트됩니다. 그럼 먼저readyState 속성 값을 CLOSED (3) 변경 작업을 큐잉하고, WebSocket 개체에서 이벤트를 발표해야 안됩니다. [WSP]

이 절에서는 큐잉되는 모든 작업에 대한 작업 소스 WebSocket 작업 소스 입니다.

5.1 Event 정의

interface CloseEvent : Event {ReadOnly attribute boolean wasClean ; ReadOnly attribute unsigned long code ; ReadOnly attribute DOMString Reason ; void initCloseEvent (in DOMString typeArg, in boolean canBubbleArg, in boolean cancelableArg, in boolean wasCleanArg, in unsigned long codeArg, in unsigned long reasonArg) ;};

initCloseEvent () 메서드는 DOM Events 인터페이스의 동명의 메서드와 마찬가지로 이벤트를 초기화해야 안됩니다. [DOMEVENTS]

wasClean 속성은 연결이 완전히 차단되었는지 여부를 나타냅니다.

code 속성은 서버가 보내온 WebSocket 연결 해제 코드를 나타냅니다.

reason 속성은 서버가 보내온 WebSocket 연결 끊기 이유를 나타냅니다.

5.2 가비지 컬렉션

이벤트 루프 작업의 실행을 시작하는 마지막 시점에서 readyState 속성 값이 CONNECTING (0) 설정된 WebSocket 객체는 open 이벤트 message 이벤트 error 이벤트, close 이벤트에 대해 등록된 이벤트 리스너가 하나라도 있으면, 가비지 컬렉션의 대상이 돼서는 없습니다.

이벤트 루프 작업의 실행을 시작하는 마지막 시점에서 readyState 속성 값이 OPEN (1) 또는CLOSING (2) 설정된 WebSocket 개체는 message 이벤트 error 이벤트, close 이벤트에 대해 등록된 이벤트 리스너가 하나라도 있으면, 가비지 컬렉션의 대상이 돼서는 없습니다.

네트워크로 전송되기 때문에 대기열에서 데이터가 설정된 연결을 가진 WebSocket 객체는 가비지 컬렉션의 대상이되고는 안됩니다. [WSP]

WebSocket 개체가 해당 연결이 열려 있는데 가비지 컬렉션의 대상이되어 버리면, 사용자 에이전트는 WebSocket 연결을 절단해야 안됩니다. [WSP]
